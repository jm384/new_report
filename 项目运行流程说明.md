# 项目运行流程说明

这份说明文档只讲项目是怎么运作的，不讲代码实现细节。文中有些地方会顺手标出程序里对应的名字，方便以后和开发人员沟通时能对上，例如“运行主目录（`run_root`）”。

## 一、这个项目到底在做什么

这个项目的目标，是围绕纽约常见的人身伤害、保险、场所责任、工地安全、滑倒摔伤等主题，自动产出一批中文法律科普文章。

它不是“一次性把文章直接写出来”，而是分成三步走：

1. 先决定写哪些主题，再去收集相关资料。
2. 再根据资料和模板，生成文章初稿。
3. 最后检查文章有没有明显问题，必要时修复，再输出最终稿。

这样做的原因很简单：

- 先找资料，能让文章不至于空泛。
- 先出初稿，再检查，能把明显的结构问题、标题问题、法律表述风险拦下来。
- 把每一步都分开保存，出了问题更容易知道卡在哪一步。

## 二、项目每次运行会产生什么

每一轮正式任务，都会有一个独立的主目录，叫运行主目录（`run_root`）。

命名方式类似：

`outputs/output_20260706_210101/`

这个时间就是这一轮开始时的时间标记。

同一轮任务的三个阶段，都会放在这个主目录下面，不会分散到多个不同的总目录里。

现在的结构是：

- `1_collect/`
- `2_generate/`
- `3_check/`

如果程序只跑到第 1 步，就只有 `1_collect/`。
如果继续跑到第 2 步，就会新增 `2_generate/`。
如果跑到第 3 步，就会新增 `3_check/`。

所以你看到一个主目录，就等于看到了这一轮任务的完整轨迹。

## 三、项目启动时，先做哪些准备

程序启动后，首先会确认几个基本条件：

1. 现在要跑哪一个阶段（`phase`）。
2. 这次是新开一轮，还是接着某个历史目录继续跑。
3. 这轮任务的主目录放在哪里。
4. 日志文件放在哪里。
5. 当前能不能正常使用大模型（LLM）。

如果你直接跑 `collect`，程序会新建一个新的主目录。

如果你单独跑 `generate` 或 `check`，程序会优先读取你指定的历史目录（`input_run_dir`）。如果你没指定，它会自动去找最近一轮可继续使用的目录。

## 四、第 1 阶段：collect 在做什么

`collect` 阶段可以理解为“选题 + 搜资料 + 整理素材 + 分析模板风格”。

它不是简单搜几个网页，而是按下面的顺序做：

### 1. 先挑出本轮要写的主题

程序会先从主题库里选出本轮主题（`selected_topics.json`）。

这些主题不是完全随机瞎选，而是从一组预设的大类里抽取，例如：

- 交通事故
- 滑倒摔伤
- 工地事故
- 公寓楼公共区域受伤
- 人行道维护责任

每个主题通常还会带一个主题类别（`category`），这主要用于后面匹配模板文档。

### 2. 再为每个主题准备搜索词

选好主题后，程序会为每个主题生成一组搜索词（`generated_queries.json`）。

这些搜索词的目的，不是追求数量多，而是尽量覆盖：

- 主题本身
- 责任判断
- 证据保存
- 保险或索赔相关信息
- 官方来源或权威来源

你可以把它理解成：程序先列出“围绕这个主题，应该去查哪些方向”。

### 3. 再去搜索候选来源

接下来程序会去搜索网页来源（候选链接）。

来源主要有几种：

- 手动预置来源（`manual_topic_urls`）
- 公开搜索结果
- 固定来源兜底搜索（例如 `nyc.gov`、`ny.gov`、`osha.gov` 等）

这里有个很重要的现实逻辑：

这个项目所在的网络环境并不总是稳定，所以公开搜索可能超时、失败，甚至抓不到结果。现在程序已经做了“快速回退”处理：

- 如果搜索连续超时，不会一直傻等下去。
- 会尽快切换到下一种搜索方式。
- 如果公开搜索整体不行，就退到预置来源和固定来源兜底。

也就是说，`collect` 的目标不是“必须每次都从搜索引擎抓一大堆新链接”，而是“尽量拿到足够可用的材料”。

### 4. 再抓取网页正文

找到候选链接以后，程序会尝试读取这些网页的正文内容。

抓下来的结果会进入中间数据，例如：

- 原始搜索结果（`raw_search_results.json`）
- 抓取后的文章数据（`extracted_articles.json`）

这一阶段的重点不是页面长得多漂亮，而是尽量拿到：

- 标题
- 链接
- 简要信息
- 页面正文（`content`）

如果程序无法精准判断哪些是正文，它会尽量保留页面里的可读文本。

### 5. 再过滤资料，但现在不过度卡死

抓取到的内容不会全部直接拿去写文章。

程序会做一轮“资料筛选”：

- 和主题明显无关的，会尽量去掉。
- 明显没内容的，会尽量去掉。
- 但不会像以前那样，因为“有效资料不足 4 篇”就直接把整个 `collect` 阶段卡死。

现在的逻辑更偏向“先交付，再提示质量”：

- 如果资料偏少，会记日志。
- 但只要还能继续，就尽量继续。

也就是说，筛选的作用现在更像“整理质量”，不是“动不动就终止任务”。

### 6. 为每个主题生成主题采集文档

每个主题都会生成一份采集文档，放在：

`1_collect/topic_research_docs/`

这份文档的作用，是让你能回头看清楚：

- 这个主题搜了什么
- 最后用了哪些链接
- 每条链接的大概内容是什么
- 页面正文保存了哪些文本

现在这类文档强调的是“便于人工回看素材”，不是技术调试。

### 7. 再准备模板文档

除了资料，程序还会为每个主题准备模板文档。

模板文档放在：

`template_docs/主题类别/多个模板文件`

也就是说，模板现在是按主题类别分目录保存的，不是所有模板堆在一个平面目录里。

程序会优先使用本地已有模板，再决定是否需要补模板。

目前模板数量有两个概念：

- 理想数量（`template_min_count`）是 4
- 最低可继续数量（`template_min_viable_count`）是 3

也就是说：

- 有 4 篇最好
- 只有 3 篇也可以继续
- 少于 3 篇才认为模板明显不够

这样做，是为了把大模型配额尽量留给后面的文章生成，而不是前面一直拿去补模板。

### 8. 最后分析模板风格

当模板准备好以后，程序会生成模板风格画像（`template_style_profiles.json`）。

这一步不是为了“抄模板”，而是为了让后面的文章更接近你想要的风格。

它会总结出一些非技术性的特征，例如：

- 文章通常怎么开头
- 段落节奏大概怎样
- 小标题通常有多少
- 结尾通常怎么收束
- 常见措辞和整体语气是什么

这就是后面写文章时的“风格参照”。

## 五、第 2 阶段：generate 在做什么

`generate` 阶段就是正式写文章。

但它不是“把主题名丢给模型，随便写一篇”，而是会同时参考三类东西：

1. 主题本身
2. `collect` 阶段保留下来的资料来源
3. 模板风格画像（`style_profile`）

### 1. 它会先读取 collect 的成果

`generate` 会去读这些关键数据：

- 选题结果（`selected_topics.json`）
- 已过滤来源（`filtered_sources.json`）
- 模板风格画像（`template_style_profiles.json`）
- 搜索词（`generated_queries.json`）

如果这些数据缺失，就不会硬写，而是会明确中止并告诉你缺了什么。

### 2. 它会要求模型按完整文章来写

生成文章时，程序现在已经加了比较严格的要求，例如：

- 第一行必须是真正的文章标题
- 不能出现“我先按你的要求”“下面我来写”这种说明腔
- 不能把“温和提醒”“古灵王律师团寄语”误当标题
- 要有完整结构，不能只给半篇
- 语言要自然，不能像提纲或命令

也就是说，`generate` 现在不只是“能出字”，而是会先卡掉一些明显不合格的写法。

### 3. 初稿写完后，还会做一次自动验收

文章生成后，程序会立刻检查几个很实际的标准：

- 标题是否像正式文章标题
- 开头有没有说明性废话
- 小标题数量够不够
- 字数够不够
- 信息展开是否明显不足

如果不过关，不会直接保存成最终初稿，而是先重试。

### 4. 先重写，再考虑补采

如果文章第一次不理想，程序会先用同一批资料重写。

如果还是不够好，就会判断是不是“素材本身偏少或偏弱”。

这时候它会自动触发“定向补采”：

- 只补当前这个主题
- 不重跑所有主题
- 尽量扩充当前主题的资料
- 新资料合并后，再重新生成

这一步的目的，是避免出现那种“文章特别短、内容发虚、结构也没展开”的情况。

### 5. 但补采不是无限循环

为了防止没完没了地重试，程序有明确上限：

- 常规重写最多 2 次
- 补采后再重写 1 次

如果到这里还不达标，就不会假装成功，而是输出 `ACTION_REQUIRED.md`，明确告诉你：

- 卡在哪个主题
- 试过什么
- 为什么还不能继续
- 你下一步该怎么处理

### 6. generate 的输出是什么

这个阶段的主要输出在：

- `2_generate/generated_blog_articles/`
- `2_generate/data/generation_metadata.json`

其中：

- `generated_blog_articles/` 是初稿文章
- `generation_metadata.json` 会记录每篇文章的关键信息，例如：
  - 标题
  - 字数
  - 是否触发补采
  - 补采了多少来源
  - 生成尝试了几次

所以 `generate` 不只是“产出文章”，也会留下这篇文章是怎么形成的过程记录。

## 六、第 3 阶段：check 在做什么

`check` 阶段不是简单打分，而是“检查并修复”。

它的目标不是给你一堆看不懂的数字，而是判断：

1. 文章结构有没有缺
2. 标题和正文是不是太僵硬
3. 阅读感觉是否接近模板风格
4. 纽约法律或保险表述有没有明显风险

### 1. 它先读取 generate 结果

它主要读取：

- `2_generate/data/generation_metadata.json`
- `1_collect/data/template_style_profiles.json`

也就是说，`check` 既看文章本身，也看它原本应该接近什么风格。

### 2. 它看什么问题

现在检查重点已经改得比较贴近你的要求，不再主打各种相似度分数。

它主要关注：

- 有没有标题、导入、小标题、收束这些基本结构
- 标题是否像正式文章标题
- 语言是否太生硬、太像说明书
- 是否反复啰嗦默认背景词
- 法律或保险表述有没有明显绝对化、误导性承诺

### 3. 如果发现问题，会先修

如果发现问题，程序不会立刻判死刑，而是先尝试修复。

修复方式包括：

- 规则替换
- 调整明显不稳妥的措辞
- 必要时调用大模型重写整篇文章

如果问题只是语言或结构，通常会直接走重写。

如果问题本质上来自“素材不够”，理论上就会回到补采再重写的思路。

### 4. 修完之后会再检查一遍

修复不是写完就算了。

程序会把修复后的文章再跑一遍同样的检查逻辑。

如果修完后还是有明显问题，就不会假装通过，而是中止并生成 `ACTION_REQUIRED.md`。

### 5. 通过后会输出终稿

如果检查通过，最终结果会放到：

`3_check/final_articles/`

这才是这一轮真正可以当成最终交付物的文章。

同时，程序还会输出：

`3_check/quality_check_report.docx`

这份报告现在主要是给人看“这一轮检查了什么、有没有触发修复、最后结论是什么”。

## 七、什么是 ACTION_REQUIRED

如果程序跑不下去了，不会静悄悄失败，而是会在本轮主目录里生成：

`ACTION_REQUIRED.md`

你可以把它理解成“人工接手提示单”。

里面通常会写清楚：

- 当前运行时间
- 当前卡在哪个阶段
- 卡在哪个主题
- 出了什么问题
- 程序已经尝试过什么
- 为什么现在不能继续
- 建议你下一步怎么做

常见原因包括：

- 大模型配额不足
- 某篇文章多次生成仍不达标
- 某个阶段缺少上一步的输入文件
- 关键依赖缺失

这个文件的意义就是：不让你猜。

## 八、项目里你最常接触的文件和目录

如果你平时主要看结果，最值得关注的是这些位置：

### 1. 主目录

例如：

`outputs/output_20260706_210101/`

这是某一整轮任务的总入口。

### 2. collect 结果

- `1_collect/data/selected_topics.json`
- `1_collect/data/generated_queries.json`
- `1_collect/data/filtered_sources.json`
- `1_collect/data/template_style_profiles.json`
- `1_collect/topic_research_docs/`

它们分别代表：

- 本轮选了什么主题
- 围绕主题搜了哪些方向
- 最终留下哪些资料
- 模板风格总结是什么
- 每个主题的采集文档是什么

### 3. generate 结果

- `2_generate/generated_blog_articles/`
- `2_generate/data/generation_metadata.json`

这代表：

- 初稿文章是什么
- 每篇文章经历了几次生成、是否补采、字数多少

### 4. check 结果

- `3_check/quality_check_report.docx`
- `3_check/final_articles/`

这代表：

- 检查报告是什么
- 最终文章是什么

### 5. 故障提示

- `ACTION_REQUIRED.md`

这代表：

- 当前这一轮为什么没法继续

## 九、如果你平时要怎么用这个项目

如果你只是作为使用者，而不是开发者，最简单的理解方式是：

### 方式 1：完整跑一轮

先跑：

`python main.py --phase collect`

然后跑：

`python main.py --phase generate --input-run-dir 某一轮主目录`

最后跑：

`python main.py --phase check --input-run-dir 同一个主目录`

这样一轮下来，你能得到：

- 采集文档
- 文章初稿
- 检查报告
- 最终文章

### 方式 2：只续跑后半段

如果 `collect` 已经跑完，不需要重头再来，可以直接指定那一轮主目录，继续跑 `generate` 或 `check`。

这就是 `input_run_dir` 的作用：告诉程序“接着哪一轮继续做”。

## 十、这个项目现在的核心运行思路

如果只用一句话概括，这个项目现在的思路是：

先收集可用素材，再按模板风格生成初稿，然后用“检查并修复”的方式尽量把文章推到可交付状态；如果还达不到要求，就明确停下来，而不是假装完成。

这也是它和很多“一次性生成文章”的工具最大的不同：

- 它不是一步出结果
- 它会记录过程
- 它会在质量不够时补采或重写
- 它会在确实不行时明确报错

## 十一、你后面判断一轮任务有没有成功，可以看什么

最简单的判断方法是看这一轮主目录里有没有这几样东西：

1. `1_collect/topic_research_docs/`
2. `2_generate/generated_blog_articles/`
3. `3_check/quality_check_report.docx`
4. `3_check/final_articles/`

如果四样都有，而且没有新的 `ACTION_REQUIRED.md` 阻塞问题，就说明这一轮基本是跑通了。

如果缺了其中某一项，就回头看：

- 是不是没跑到那个阶段
- 还是那个阶段中途被 `ACTION_REQUIRED.md` 拦住了

## 十二、最后给你的使用建议

如果你不是程序员，平时最实用的工作方式其实很简单：

1. 先看本轮主目录。
2. 再看 `topic_research_docs`，确认素材方向对不对。
3. 再看 `generated_blog_articles`，确认初稿是否像正常文章。
4. 最后看 `final_articles`，把这里当成最终交付版本。
5. 如果出了问题，优先看 `ACTION_REQUIRED.md`，不要先猜。

- 每个目录应该怎么看
- 发现什么情况算正常
- 发现什么情况需要重跑
- 哪些文件最值得人工抽查
