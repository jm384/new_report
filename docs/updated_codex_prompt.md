# 纽约州华人律师事务所中文法律科普博客自动生成项目 Codex 提示词（更新版）

## 目标

开发一个完整、可运行、结构清晰的 Python 项目，用于自动生成中文法律科普博客，并输出真实可用的 `docx` 文件。

## 当前确认需求

1. 项目分为 `collect`、`check`、`generate` 三个阶段。
2. 输出目录统一使用单主目录结构：`outputs/output_YYYYMMDD_HHMMSS/`。
3. 主目录下按阶段创建子目录：
   - `1_collect/`
   - `2_check/`
   - `3_generate/`
4. 运行到哪个阶段，就创建哪个子目录；未运行到的阶段不创建。
5. `check` 阶段必须检查、修复，并输出最终文章到 `2_check/final_articles/`。
6. 检查报告只保留 `docx`，不再保留 `md`。
7. 检查逻辑只关注四类问题：
   - 文章结构是否遗漏
   - 标题与内容是否过于僵硬
   - 文章整体阅读感觉与模板是否接近
   - 法律/保险相关表述是否存在明显风险
8. `collect` 阶段默认不因“有效资料不足”而终止，质量检查只用于提示，不作为硬中断。
9. `template_docs/` 必须采用“主题类别目录 -> 多个模板文件”的结构。
10. 模板数量理想目标为 4 篇，最低可行值为 3 篇。
11. 当模板只有 3 篇时，允许继续分析，但要在日志和报告中说明。
12. 模板不足时优先节省 LLM 配额，把配额留给 `generate`。
13. `generate` 阶段仍必须依赖 LLM，不允许伪造文章结果。
14. 如果 LLM 配额不足，应输出明确的 `ACTION_REQUIRED.md`，说明是“配额不足”。

## 文章输出要求

1. 输出必须是原生 `docx` 格式，不得把 Markdown 语法直接写进文章。
2. 文章应有真实标题、分段、小标题和结尾提醒。
3. 内容虽然面向华人读者，但正文中不要频繁提及“华人”。
4. 默认场景就是纽约，因此正文中不要过度重复“纽约”。
5. 文风要更自然、顺畅、像真实律师事务所博客，而不是硬模板文。

## 采集文档要求

1. 主题采集文档只保留：
   - 搜索词
   - 采集链接
   - 链接简单概要
2. 如果使用了相关性评分，可以保留，但筛选标准不要设得过高。

## 文档同步要求

1. README 的目录结构和命令示例必须与实际 CLI 一致。
2. 所有中文日志、提示语、`ACTION_REQUIRED.md` 和报告内容都必须正常显示。
3. 这份更新版提示词文档要长期保存，便于后续继续迭代。
