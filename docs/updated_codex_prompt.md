# 纽约州华人律师事务所中文法律科普博客自动生成项目 Codex 提示词（更新版）

## 目标

开发一个完整、可运行、结构清晰的 Python 项目，用于自动生成中文法律科普博客文章，并输出真实可用的 `docx` 文件。

## 当前确认的需求

1. 项目分为 `collect`、`generate`、`check` 三个阶段。
2. 输出目录使用单主目录结构：`outputs/output_YYYYMMDD_HHMMSS/`。
3. 主目录下按阶段生成子目录：
   - `collect/`
   - `generate/`
   - `check/`
4. 程序运行到哪一阶段，才创建对应子目录；未运行到则不创建。
5. `check` 阶段除评分外，还要自动修复文章，并输出最终文章到 `check/final_articles/`。
6. 检查报告需要输出 `md`、`docx`、`json` 三种格式。
7. 评分项必须有通俗解释，尤其是“整体相似度”等字段。
8. `collect` 阶段默认不因“有效资料不足”而终止，质量检查只用于提示。
9. 主题采集文档中必须包含采集源数据，不只是摘要。
10. `template_docs/` 必须采用“主题类别目录 -> 多个模板文件”的结构。
11. 模板理想目标为 4 篇，最低可行值为 3 篇。
12. 当模板只有 3 篇时，允许继续分析，但需要记录说明。
13. 模板数量不足时应优先节省 LLM 配额，把配额留给 `generate`。
14. `generate` 阶段仍必须依赖 LLM，不允许伪造文章结果。
15. 如遇 LLM 配额不足，应生成明确的 `ACTION_REQUIRED.md`，说明是“配额不足”。

## 文章输出要求

1. 输出必须是原生 `docx` 格式，不得把 Markdown 语法直接写入文章。
2. 文中序号要用真实序号结构，重点内容要用真实加粗，而不是 `#`、`-`、`**` 这类 Markdown 标记。
3. 内容虽然面向华人读者，但正文中不要频繁提及“华人”。
4. 默认场景就是纽约，因此正文中不要过度重复“纽约”。
5. 文风需要更自然、更通顺、更像真实律所博客，而不是生硬模板文。

## 检查阶段要求

1. 不仅要评分，还要修复文章。
2. 修复后要输出最终文章。
3. 报告中要写清楚评分说明。
4. 需要保留原文路径和最终文路径，方便比对。

## 采集阶段要求

1. 搜索词、原始结果、过滤结果、采集文章、模板风格分析都要保存。
2. 主题采集文档中需要展示：
   - 搜索词
   - 采集链接清单
   - 素材要点
   - 采集源数据摘录
3. 资料不足时以警告记录为主，不因默认阈值直接终止。

## 文档同步要求

1. README 中的目录结构和命令示例必须与实际 CLI 一致。
2. 所有中文日志、提示语、`ACTION_REQUIRED.md`、报告内容都要正常显示，不能乱码。
3. 这份更新版提示词需要长期保存在仓库中，便于后续查看和继续沟通。
